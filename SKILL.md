# academic-search - 学术检索与文献总结

本 Skill 只做一件事：当用户需要“查论文、做文献综述”时，用已有的 `web_search` 和 `web_fetch`，针对学术站点做精细搜索与结构化总结。

> 安全原则：
> - 只把 **检索关键词** 发到搜索引擎，不包含用户的名字、帐号、IP、VPS 信息或合同内容。
> - 不把完整的技术协议、隐私对话原样丢给外部接口；只在必要时提取几个关键词（英文）作为检索词。
> - 只调用 OpenClaw 自带的 `web_search` / `web_fetch`，不在服务器上安装任何额外二进制程序或跑未知脚本。

## 什么时候使用本 Skill

满足这些条件时触发：

- 用户显式提到：
  - “帮我搜一下关于 X 的论文/文献/研究”
  - “谷歌学术 找 X”
  - “做一个 X 方向的文献综述/研究进展汇总”
  - “找几篇代表性论文，对比一下结论/方法”
- 或者问题明显是“需要查学术资料”而不是普通网页：
  - 领域关键词 + “paper / journal / conference / IEEE / arXiv / Nature / Science 等”

**不要使用本 Skill 的情况：**

- 用户只问基础概念、教科书级别内容（现有模型知识足够回答时）；
- 用户只是想要通俗解释，而不是“按论文分条引用”；
- 需要访问登录后资源（机构订阅期刊、CNKI 全文等）——这些后端目前不接入，只能用公开摘要。

## 检索策略

### 1. 构造检索词（中文 → 英文）

当用户用中文描述主题时：

1. 在本地先用模型把主题翻成简洁的英文关键词组合（2–6 个词），例如：
   - “机器人焊接轨迹优化” → `robotic welding path planning optimization`
   - “大电流脉冲焊接热影响区” → `high current pulsed welding heat affected zone`
2. 根据需求补上限定词：
   - 要最近几年 → 加 `2019..2025` 或 `"2020" OR "2021" ...`
   - 需要方法类 → 加 `method` / `approach`
   - 需要对比研究 → 加 `comparison` / `benchmark`

始终避免把 **用户身份、项目名称、服务器 IP、合同细节** 写进检索词。

### 2. 站点优先级

按下列优先级组合使用 `web_search`：

1. **arXiv**（预印本，开放）：
   - 查询：`<英文关键词> site:arxiv.org`
2. **Google Scholar 页面（若可索引到）**：
   - 查询：`<英文关键词> site:scholar.google.com`
   - 只用于抓取题目/作者/年份/引用信息，不依赖登录态。
3. **出版社 / 期刊官网**：常见域名：
   - `ieeexplore.ieee.org`
   - `dl.acm.org`
   - `springer.com` / `link.springer.com`
   - `sciencedirect.com`
   - `tandfonline.com`
   - `wiley.com`
   - 对于焊接等工程方向，可额外尝试 `site:asme.org`、`site:aws.org` 等（按需要扩展）。

根据场景可以多次调用 `web_search`，每次 `count` 控制在 3–5 条，综合去重后再挑最有价值的若干篇。

## 抓取与摘要流程

对于每一条候选论文链接：

1. 使用 `web_fetch(url, extractMode="text" 或 "markdown")` 拉取正文：
   - 若页面较长，限制 `maxChars`（例如 8000–12000 字），避免塞爆上下文；
   - 优先抓到摘要区域（Abstract），如果能简单定位到 `Abstract` 标题，可只截取摘要上下的部分。
2. 从抓取到的文本中提取关键信息：
   - 标题（Title）
   - 作者（Authors）
   - 年份（Year）
   - 发表渠道（Journal / Conference / Preprint）
   - 摘要（核心研究问题 + 方法 + 结果）
3. 用模型在本地总结成**中文条目**，结构建议统一为：

   ```text
   1）标题（英文原文）
      - 作者 / 年份 / 来源：...
      - 研究问题：...
      - 方法：...
      - 主要结论：...
      - 局限 / 注意点：...
      - 链接：<原文链接>
   ```

4. 多篇论文时，再做一层综合：
   - 共性结论 / 主流观点
   - 不同方法的优缺点
   - 尚未解决的问题或未来方向

## 隐私与安全约束

在使用本 Skill 时，必须遵守：

1. **只发送必要的检索信息**
   - 仅使用经过“压缩”的学术主题关键词；
   - 不要把：
     - 用户的名字、Telegram 账号、邮箱；
     - VPS IP、端口、域名；
     - 具体技术协议条款、项目代号、内部文件名；
     - 任何能将请求关联到现实身份/基础设施的标识符；
     放到 `web_search` 或 `web_fetch` 的参数里。

2. **不调用非 OpenClaw 管理的外部可执行文件**
   - 不在服务器上下载、安装第三方软件、脚本或二进制来“辅助搜索”；
   - 不执行远程来源的 shell 脚本、一键安装脚本等。

3. **只用 HTTP GET 获取公开网页**
   - 不提交表单 / 不登录 / 不上传文件到外部站点；
   - 只访问公开可见的论文摘要、预印本、索引页。

## 输出风格约定

当本 Skill 被触发时，对用户的回答请遵守：

1. **先给结论，再给细节**
   - 开头简要说明：
     - 找到了大约多少篇相关论文；
     - 总体研究趋势 / 主流结论；
   - 然后按编号列出 3–8 篇代表性文献。

2. **中文总结 + 保留英文关键信息**
   - 标题用英文原文为主，可在括号里给简短中文意译；
   - 作者用英文列出前 1–3 位即可；
   - 重要术语（模型名、算法名）用原文保留。

3. **给出原文链接，便于后续自行查阅**
   - 每篇都附上至少一个可访问的 URL；
   - 如果有 arXiv / 预印本链接，优先给这个。

4. **如果检索结果不多或质量一般，要实话实说**
   - 说明“公开可检索文献较少”，不要强行凑数；
   - 不编造论文标题或引用信息。

## 失败与降级处理

- 如果 `web_search` 在多个站点上都返回很少结果：
  - 明确告诉用户目前公开文献稀少；
  - 可以退而求其次，搜索相关领域（更宽泛关键词），说明这是“相关但不完全匹配”的工作。

- 如果 `web_fetch` 频繁失败 / 403 / 需要登录：
  - 不要尝试模拟登录或绕过限制；
  - 直接提示“该出版社页需要订阅/登录，只能基于摘要/其他来源给出有限总结”。
